# TrendBird 実装計画書作成ガイド v1.0

最終更新: 2026-02-21

---

## 1. 概要・目的

実装計画書は **実装の設計図** である。
Phase 分割された計画書により、実装の順序・依存関係・進捗を明確に管理する。

### 核心ルール

1. **計画セッションでは実装禁止**: 実装計画書の作成が目的のセッションでは、絶対にコードの作成・修正を行わない。`docs/` にファイルを保存して完了とする
2. **チェックボックス更新義務**: 実装計画書に従って実装を行ったセッションでは、完了したタスクのチェックボックスにチェックを入れて計画書を更新する
3. **1計画書1ファイル**: 計画書は `docs/` 配下に1ファイルとして保存する

### 原則

- 計画書を読むだけで、何を・どの順序で・どのように実装するかが分かる状態を目指す
- Phase 間の依存関係を明示し、並行作業可能な Phase を特定する
- 各タスクは十分に具体的で、実装者が迷わず着手できる粒度にする
- 参照すべきスキル（デザインシステム、アーキテクチャガイド等）を Phase ごとに明示する

## 2. ファイル命名規則

### 形式

```
docs/{NN}-{kebab-case}.md
```

- `{NN}`: 2桁の連番（`01`, `02`, ...）
- `{kebab-case}`: 計画書の内容を表す英語のケバブケース

### 採番手順

1. `docs/` 配下の既存ファイルを確認する
2. `{NN}-` prefix が付いたファイルが存在する場合、最大の連番 + 1 を使用する
3. `{NN}-` prefix が付いたファイルが存在しない場合、`01` から開始する

### 具体例

```
docs/01-backend-implementation-plan.md
docs/02-frontend-dashboard-plan.md
docs/03-auth-flow-integration-plan.md
```

### 既存ファイルのリネーム

`docs/` に prefix なしの計画書ファイルが既に存在する場合:
- **自動リネームは行わない**
- ユーザーに「既存の `{ファイル名}` を `{NN}-{ファイル名}` にリネームしますか？」と確認する
- ユーザーが承認した場合のみリネームを実行する

## 3. 計画書テンプレート

以下のテンプレートに従って計画書を作成する。

````markdown
# {タイトル}

> **前提**: {この計画書の前提条件}
> **目標**: {この計画書で達成する最終的な目標}
> **作成日**: YYYY-MM-DD
> **ステータス**: 未着手 | 進行中 | 完了

---

## 依存グラフ

```
Phase 0: {名前}
    │
    ▼
Phase 1: {名前}
    │
    ├──────────┐
    ▼          ▼
Phase 2      Phase 3  (並行可能)
    │          │
    ...
```

> {並行作業可能な Phase の補足説明}

---

## Phase 0: {Phase名}

### 目的

{この Phase で達成すること}

### 参照スキル

- `.claude/skills/{skill-name}/SKILL.md`

### タスク

- [ ] **{タスク名}** — {具体的な説明}
- [ ] **{タスク名}** — {具体的な説明}

### 作成ファイル

| # | ファイルパス | 責務 |
|---|------------|------|
| 1 | `path/to/file.ext` | {このファイルの責務} |

### 検証方法

- {検証手順1}
- {検証手順2}

---

## Phase N: {Phase名}

{同じ構造を繰り返す}

---

## 全Phase進捗サマリー

| Phase | 内容 | タスク数 | 完了数 | ステータス |
|-------|------|---------|--------|-----------|
| 0 | {Phase名} | {N} | {N} | 未着手 / 進行中 / 完了 |
| 1 | {Phase名} | {N} | {N} | 未着手 / 進行中 / 完了 |
| ... | ... | ... | ... | ... |

## 参照スキルマトリクス

| Phase | {skill-1} | {skill-2} | {skill-3} |
|-------|:---------:|:---------:|:---------:|
| 0 | x | | |
| 1 | x | x | |
| ... | ... | ... | ... |
````

### テンプレートの補足

- **依存グラフ**: ASCII アートで Phase 間の依存関係を視覚化する。並行可能な Phase は横に並べる
- **Phase の順序**: 依存関係に基づいて番号を振る（0 始まり）
- **参照スキル**: その Phase の実装で参照すべきスキルファイルを列挙する。参照すべきスキルがない場合はセクションを省略可
- **作成ファイル**: その Phase で新規作成・修正するファイルを表形式で列挙する。修正のみの場合はセクションを省略可
- **検証方法**: ビルドコマンド、テスト、手動確認など具体的な検証手順を記述する
- **全Phase進捗サマリー**: 計画書全体の進捗を一覧できるテーブル。実装セッションで更新する
- **参照スキルマトリクス**: どの Phase でどのスキルを参照するかを俯瞰できるテーブル

## 4. チェックボックス運用ルール

### フォーマット

- 未完了: `- [ ] **タスク名** — 説明`
- 完了済: `- [x] **タスク名** — 説明`

### 粒度の目安

- 1タスク = 1つの実装セッション（またはそのサブセット）で完了できる単位
- 目安: 1〜3ファイルの作成・修正に対応する粒度
- タスクが大きすぎる場合はサブタスクに分割する

### 実装セッションでの更新手順

1. 実装を開始する前に、対象 Phase のタスクを確認する
2. 実装が完了したタスクのチェックボックスを `- [x]` に更新する
3. 全Phase進捗サマリーの「完了数」と「ステータス」を更新する
4. Phase 内の全タスクが完了したら、ステータスを「完了」に変更する

### 注意事項

- チェックボックスは実装が **検証を含めて完了** した場合にのみチェックする
- ビルドエラーやテスト失敗がある状態ではチェックしない
- 計画書にないタスクを実装した場合は、対応する Phase にタスクを追加してからチェックする

## 5. 計画書作成セッションのワークフロー

### 禁止事項

- コードファイル（`.go`, `.ts`, `.tsx`, `.css` 等）の作成・修正
- `go build`, `npm run build` 等のビルドコマンドの実行
- テストの実行
- パッケージのインストール

### 許可操作

- 既存コードの **読み取り**（Read, Glob, Grep）— 設計判断の材料として
- `docs/` 配下へのマークダウンファイルの作成
- 既存の計画書ファイルの更新
- 既存スキルファイルの読み取り（参照スキルの確認）

### 作成手順

1. **要件ヒアリング**: ユーザーの要求を確認し、不明点があれば質問する
2. **既存コード調査**: 関連する既存コード・型定義・スキルを読み取り、現状を把握する
3. **依存関係の整理**: 実装すべき機能の依存関係を分析し、Phase 分割を設計する
4. **依存グラフ作成**: ASCII アートで Phase 間の依存関係を視覚化する
5. **Phase 詳細の記述**: 各 Phase のタスク・作成ファイル・検証方法を具体的に記述する
6. **進捗サマリー作成**: 全Phase進捗サマリーテーブルを作成する
7. **参照スキルマトリクス作成**: Phase ごとの参照スキルを整理する
8. **ファイル保存**: 命名規則に従って `docs/` に保存する

### セッション終了時の報告フォーマット

計画書の作成が完了したら、以下の形式でユーザーに報告する:

```
計画書を作成しました。

- ファイル: `docs/{NN}-{name}.md`
- Phase数: {N}
- 総タスク数: {N}
- 並行可能な Phase: {Phase X と Phase Y}
```

## 6. 品質チェックリスト

計画書を `docs/` に保存する前に、以下を確認する。

### 構造

- [ ] YAML frontmatter 相当のメタ情報（前提・目標・作成日・ステータス）があるか
- [ ] 依存グラフが記述されているか
- [ ] 全 Phase に「目的」「タスク」「検証方法」があるか
- [ ] 全Phase進捗サマリーテーブルがあるか
- [ ] 参照スキルマトリクスがあるか

### タスク

- [ ] すべてのタスクにチェックボックス（`- [ ]`）が付いているか
- [ ] タスクの粒度が適切か（1タスク = 1〜3ファイル程度）
- [ ] タスク名が太字（`**タスク名**`）で記述されているか
- [ ] タスクの説明が具体的で、実装者が迷わず着手できるか

### 依存関係

- [ ] Phase の番号が依存関係の順序と整合しているか
- [ ] 並行可能な Phase が依存グラフで明示されているか
- [ ] 各 Phase の依存先 Phase が明記されているか（依存がある場合）

### 参照スキル

- [ ] 各 Phase で参照すべきスキルが漏れなく記載されているか
- [ ] スキルファイルのパスが正しいか（`.claude/skills/{name}/SKILL.md`）

### 命名

- [ ] ファイル名が `{NN}-{kebab-case}.md` 形式か
- [ ] 連番が正しく採番されているか
