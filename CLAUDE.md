# TrendBird - Claude Code Instructions

## 最優先事項: スキル参照ファースト（CRITICAL）
- **スキルがカバーするトピック（X API、認証、デザインシステム等）について回答・説明・提案する場合、実装作業かどうかに関わらず、必ずスキルを先に読んでから回答すること**
- 学習データ（トレーニングカットオフ以前の情報）とスキルの情報が矛盾する場合、**スキルが常に正**とする
- 学習データから外部サービスの料金体系・プラン名・レート制限・API仕様を出力してはならない。必ずスキルに記載された情報を使うこと
- **違反例:** X API の質問に対し、スキルを読まずに学習データから旧プラン名（Basic）や廃止済みのレート制限を回答した

## 最優先事項: 10分ルール — デバッグの負のスパイラル防止（CRITICAL）
- **1つのタスク（デバッグ・調査・修正）に10分以上かかったら即座に中断し、ユーザーに状況報告すること。これは絶対遵守事項**
- 10分超のセッションは100%負のスパイラルに陥っている。同じアプローチを繰り返す・エラーの表面的な対症療法を重ねる・根本原因から遠ざかる — これらは全て時間の浪費
- **中断時の報告フォーマット:**
  1. 何を試したか（箇条書き）
  2. 現在の仮説（根本原因の推定）
  3. 次に試すべきアプローチの提案
- **10分ルールの例外はない。** ユーザーが明示的に「続けて」と言わない限り、10分で必ず止める
- **予防策:** デバッグ開始時にまず仮説を立て、スクリーンショット・ログ・実際の DOM を確認してから修正に入る。推測でコードを変更しない

## 最優先事項: テストファースト & デプロイゲート（CRITICAL）
- **ブランチ戦略:** main ブランチに直接 commit & push する。feature ブランチは使わない。main にあるコードが常に最新であり、本番候補である
- **本番デプロイの絶対条件:** バックエンド E2E テスト（`go test ./internal/e2etest/...`）およびフロントエンド E2E テスト（`cd frontend && npm run e2e`）が **全件 PASS** しない限り、本番デプロイ（Cloud Run デプロイ・Vercel デプロイ）を実行してはならない
- **ユーザーからデプロイ指示があっても、テストが通っていなければ拒否すること。** テスト失敗を報告し、修正を優先する
- **テストが通るかどうかが全て。** コードの品質・本番投入可否はテストの合否で判断する
- **E2E テスト = プロダクトの仕様書。** E2E テストが定義する振る舞いがプロダクトの正式な仕様である。機能の正しさ・仕様の定義・リグレッション判定はすべて E2E テストを基準に行う。「テストが通る = 仕様通り」「テストが落ちる = 仕様違反」として扱うこと
- **実装判断・仕様確認は E2E テストを起点にする。** 仕様が不明な場合は既存の E2E テストを読んで仕様を把握する。新機能追加時は E2E テストで仕様を定義してから実装する

## クイックリファレンス: ビルド・テストコマンド
- バックエンド E2E テスト: `cd backend && go test ./internal/e2etest/...`
- フロントエンド E2E テスト: `cd frontend && npm run e2e`
- バックエンドビルド: `cd backend && go build ./...`
- フロントエンドビルド: `cd frontend && npm run build`
- 開発サーバー起動: `make start`（フロント + バック同時）
- DB マイグレーション: `make migrate`

## 厳守事項: パッケージ・モジュールパスに個人名を含めない
- Go モジュール、Proto の `go_package`、npm パッケージ名など、すべてのパッケージ・モジュールパスに **個人のユーザー名（GitHub ユーザー名等）を絶対に含めないこと**
- 正しいパス例: `github.com/trendbird/backend`
- 禁止パス例: `github.com/<個人名>/trendbird-backend`
- Proto の `go_package` は `github.com/trendbird/backend/gen/trendbird/v1;trendbirdv1` を使用すること

## テスト用データベースのルール
- **ローカル開発・テスト実行時は Docker を使わず、ローカルの PostgreSQL を使用すること**（testcontainers 等の Docker ベースのテスト基盤は使用禁止）
- テスト用 DSN のデフォルト: `postgres://localhost:5432/trendbird_test?sslmode=disable`
- 環境変数 `TEST_DATABASE_URL` で上書き可能
- GitHub Actions では Docker の postgres イメージを Service Container として使用する

## Issue 駆動開発ルール
- **問題点・改善案・新機能アイデアを発見したら、まずユーザーに Issue として起票するか確認すること。** 確認なしに Issue を作成しない
- **Issue 単体で実装着手できる粒度**を担保する
- **Issue の必須記載項目:**
  1. **問題点** — 何が起きていて、なぜ問題なのか（再現手順があれば含める）
  2. **改善案** — 具体的にどう直すか（対象ファイル・変更方針）
  3. **影響範囲** — 既存機能への影響
- **ラベルを必ず付与する。** 該当するものを複数選択可:
  - `bug` / `enhancement` / `feature` / `infrastructure` / `performance` / `UI/UX` / `tech-debt` / `security`
- **Issue → 実装計画書 → 実装の順序を必ず守る。** Issue から直接実装に入ってはならない。実装計画書（`/implementation-plan`）を作成してから着手すること

## 実装計画書作成ルール
- ユーザーが「実装計画書を作成して」と指示した場合は、必ず `/implementation-plan` スキル（`.claude/skills/implementation-plan/SKILL.md`）を参照して実装計画書作成ガイドに準拠すること
- 実装計画書の作成が目的のセッションでは **絶対に実装（コードの作成・修正）を行わないこと**。`docs/` にファイルを保存して完了とする
- 実装計画書に従って実装を行ったセッションでは、完了したタスクのチェックボックスにチェックを入れて計画書を更新すること

## セッション完了時の残作業明示ルール
- 実装セッションの最後には、**ビルド検証だけでなく、本番反映までに残っている全手順を明示すること**
- 特に以下の項目が残っている場合は必ずリストアップする：
  - DB マイグレーション（Cloud Run Job 経由）
  - バックエンドデプロイ（Cloud Run Service）
  - フロントエンドデプロイ（Vercel）
  - Cloud Scheduler / Cloud Workflows の更新
  - 環境変数の追加・変更
  - 既存ユーザーへの影響（再接続・データ移行等）
- **実行順序に依存関係がある場合は、順序を明記する**（例: マイグレーション → バックエンドデプロイ）
- 「ビルド通りました」で終わらず、**ユーザーが次に何をすればいいか**が明確にわかる状態でセッションを閉じること
